# Scalability Assessment: Rebar from Solo Swarms to 1000-Dev Enterprise

**From:** Office 180 + blindpipe adoption experience
**Date:** 2026-03-20

---

## Executive Summary

Rebar's core abstractions (contracts, Cold Start Quad, testing cascade, role-based agents, two-tag system) are correct at any scale. The philosophy doesn't change — the substrate evolves from files to services as the organization grows. The critical design goal: **a solo developer with 2 repos should feel zero overhead, while an enterprise with 500 repos gets governance, observability, and cross-team coordination — and the transition between them is gradual, never a rewrite.**

This document maps the adoption curve from solo → small team → department → enterprise, identifies what breaks at each transition, and proposes how rebar evolves to handle it.

---

## The Adoption Curve

### Tier 1: Solo / Swarm (1-3 devs, 1-5 repos)

**This is rebar's sweet spot today.** Zero infrastructure. Everything is files + grep + bash.

**What you use:**
- Cold Start Quad (README, QUICKCONTEXT, TODO, AGENTS, CLAUDE.md)
- Contracts in `architecture/` with `CONTRACT:` headers in source
- ASK CLI for role-based agent queries (persistent sessions, local)
- Steward for quality scans (local, on-demand)
- `check-todos.sh`, `check-ground-truth.sh` as pre-commit hooks
- `link:` for cross-repo dependencies (repos are siblings on disk)
- METRICS file maintained manually

**What you DON'T need:**
- CI/CD integration (you push when it's ready)
- Central catalog (you know all your repos)
- Package registry (link: works fine)
- Governance (you ARE the governance)
- Dashboard (steward output is right there)

**Overhead:** ~30 minutes to scaffold a new repo. 5-minute cold start per session. Pre-commit hooks run in seconds. This is less overhead than setting up ESLint.

**The feel:** Light. Fast. You're writing contracts because they help *you* — not because someone mandated it. ASK saves you tokens. Steward catches drift you'd otherwise miss. The two-tag system is just grep.

### Tier 2: Small Team (4-15 devs, 3-15 repos)

**The first pressure point: shared state.**

**What changes:**
- Multiple people edit QUICKCONTEXT.md and TODO.md → merge conflicts
- Someone needs to own contract reviews (not just "whoever gets to it")
- New team members need onboarding (can't just say "read DESIGN.md")
- CI becomes important (not just local pre-commit hooks)
- Cross-repo changes need coordination (PR in repo A depends on PR in repo B)

**What you add:**

| Addition | Purpose | Effort |
|----------|---------|--------|
| **CI runs steward + enforcement scripts** | Quality gates on every push | 1 hour (GitHub Actions / GitLab CI) |
| **CODEOWNERS for `architecture/`** | Contract changes need review | 5 minutes |
| **PR template referencing contracts** | "Which contracts does this touch?" | 10 minutes |
| **Computed METRICS** | Eliminate merge conflicts on counts | 1 hour (script change) |
| **Team standup via ASK** | `ask steward summary` in Slack | 15 minutes |

**What evolves:**

**METRICS becomes computed, not maintained.** `check-ground-truth.sh` already knows the real counts. Flip the flow: the script *generates* METRICS (or a CI artifact), rather than verifying a manually-maintained file. No more merge conflicts on counts.

```bash
# Before (Tier 1): manual file, script verifies
scripts/check-ground-truth.sh  # "DRIFT: test_files says 44, code says 46"

# After (Tier 2): script generates, CI publishes
scripts/compute-metrics.sh > METRICS  # always accurate
```

**QUICKCONTEXT.md gets a generated section.** Keep the human-written "What's happening" section, but auto-generate the metrics table from git + CI:

```markdown
## Current State
<!-- AUTO-GENERATED by CI — do not edit below this line -->
| Metric | Value | Updated |
|--------|-------|---------|
| Tests passing | 847/850 | 2026-03-20 08:14 |
| Steward health | 8/10 contracts VERIFIED | 2026-03-20 08:14 |
| Branch | main | — |
<!-- END AUTO-GENERATED -->

## What's Happening (human-written)
Working on offline auth flow. Alex is on the sidebar refactor.
```

**TODO.md coexists with your issue tracker.** Don't replace Jira/Linear — feed it. TODO.md becomes the agent-readable view of your backlog. A simple script syncs labels or statuses. Agents read TODO.md; humans use whatever tool they prefer.

**The feel:** Still light. CI does the boring enforcement. CODEOWNERS prevents accidental contract changes. New devs read the Cold Start Quad and are productive in 30 minutes. The overhead is ~1 day of setup, then it's invisible.

### Tier 3: Department (15-50 devs, 10-40 repos)

**The second pressure point: discovery and coordination.**

With 40 repos, you can't hold the full picture in your head. "Which repo owns the auth contract?" "Who depends on our blobstore interface?" "Is anyone else building an offline sync library?"

**What changes:**
- Cross-repo grep is painful (40 repos × clone × grep)
- Contract duplication risk (two teams build similar things independently)
- Breaking contract changes affect teams you've never talked to
- ASK agent knowledge is siloed per developer's machine
- New hires can't find anything ("where is the auth contract?")
- Package distribution via `link:` is impractical (not everyone has every repo)

**What you add:**

| Addition | Purpose | Effort |
|----------|---------|--------|
| **Contract catalog** | Searchable index across all repos | 1-2 weeks |
| **Private package registry** | Distribute shared packages without `link:` | 1 day (Verdaccio / GitLab Packages / Artifactory) |
| **Shared ASK memory** | Agent knowledge persists across developers | 1 week |
| **RFC process for new contracts** | Prevent duplication, coordinate interfaces | Process, not code |
| **Deprecation policy** | Contracts aren't abandoned overnight | Policy document |
| **Progressive onboarding** | 2-min quickstart → 10-min orientation → full methodology | 1 day of writing |

**The Contract Catalog:**

This is the keystone addition at Tier 3. Every repo's CI already runs `steward.sh` and produces JSON in `architecture/.state/`. The catalog collects these:

```
Each repo's CI:
  steward.sh → architecture/.state/steward-report.json → push to catalog

Catalog (simple service or even a static site):
  - Aggregates all steward reports
  - Indexes all contracts across all repos
  - Shows dependency graph (who depends on what)
  - Flags breaking changes (contract version bumped → find all dependents)
  - Search: "find all contracts tagged 'auth'" across 40 repos
```

**The catalog is just a git repo.** No services. No infrastructure. 3 devs coordinating 6 projects don't need a dashboard — they need a shared index they can grep.

**Implementation: `contract-catalog` git repo**

```
contract-catalog/           ← shared git repo, nothing else
├── reports/
│   ├── blindpipe.json      ← steward-report.json from blindpipe
│   ├── office180.json      ← steward-report.json from office180
│   ├── opendockit.json
│   └── tdflite.json
├── index.md                ← auto-generated: all contracts across all repos
├── deps.md                 ← auto-generated: dependency graph
└── scripts/
    ├── collect.sh           ← pulls reports from sibling repos
    └── build-index.sh       ← generates index.md + deps.md from JSON
```

**Workflow (no CI needed, no services, just git):**

```bash
# Developer runs this when they want the cross-repo picture
cd contract-catalog
./scripts/collect.sh        # copies steward-report.json from each sibling repo
./scripts/build-index.sh    # regenerates index.md and deps.md
git add . && git commit -m "update catalog"
git push

# Or just: ask architect "what contracts does office180 depend on from blindpipe?"
# The architect agent reads the catalog repo
```

**If you later grow to 50+ repos**, this same repo can get CI (auto-collect on push to any project repo) and a GitHub Pages site built from index.md. But for 3 devs and 6 repos, it's just a shared git repo with some shell scripts. **Git is the only infrastructure.**

The key insight: **the steward-report.json format is already designed for aggregation.** Rebar doesn't need to add a catalog service — it needs to provide `collect.sh` and `build-index.sh` and document the pattern.

**Package Dependencies: `link:` stays default, registries are optional.**

For 3 devs with 6 repos on their machines, `link:` is perfect. Everyone has the repos as siblings. `pnpm install` resolves locally. No network needed. This doesn't need to change until:
- You have developers who don't have all repos cloned, OR
- You have CI runners that need dependencies

When that day comes, add a private registry *alongside* `link:`, not instead of it:

```jsonc
// package.json — registry version for CI and new devs
"@blindpipe/classification-ui": "^0.1.0"

// .pnpmfile.cjs — local override for active cross-repo devs
function readPackage(pkg) {
  if (pkg.dependencies?.['@blindpipe/classification-ui']) {
    pkg.dependencies['@blindpipe/classification-ui'] = 'link:../blindpipe/packages/classification-ui';
  }
  return pkg;
}
```

But **don't add the registry until you actually need it.** For a small team, `link:` is lighter, faster, works offline, and has zero infrastructure.

**Shared ASK Memory: just un-gitignore it.**

At Tier 3, the architect agent should remember what it told developer A when developer B asks a related question. The simplest evolution:

```
Tier 1-2: agents/architect/memory.md → gitignored (local, per-machine)
Tier 3:   agents/architect/memory.md → tracked in git (shared across team)
```

The memory.md file is currently gitignored. At Tier 3, you **un-ignore it** so agent knowledge accumulates across the team and is shared via git push/pull. The memory.log.md (append-only history) becomes a team knowledge base.

The ASK CLI doesn't change. The storage doesn't change. You just start committing the file. That's it.

**The feel:** Still just git. No new infrastructure. The catalog is a git repo you grep. Packages resolve via filesystem links. Agent memory is a committed markdown file. A developer working on one repo still feels Tier 1 — the cross-repo tools are there when you need them but invisible when you don't.

### Tier 4: Enterprise (50-1000+ devs, 40-500+ repos)

**The third pressure point: governance, compliance, and organizational trust.**

At enterprise scale, the question isn't "does this work technically?" — it's "can we trust that 200 teams are following the methodology, that contracts are honored, that breaking changes are communicated, and that security-critical interfaces are reviewed?"

**What changes:**
- Contract changes can break dozens of teams you've never heard of
- Compliance requirements (SOC2, FedRAMP) need evidence of process
- Security-critical contracts need mandatory review gates
- Teams have different maturity levels (some teams are Tier 1, some are Tier 3)
- Namespace collisions become real (two teams pick the same contract ID)
- Agent autonomy needs organizational boundaries (an agent shouldn't push to another team's repo)

**What you add:**

| Addition | Purpose | Effort |
|----------|---------|--------|
| **Contract catalog as a service** | Real-time search, dependency graph, impact analysis | 2-4 weeks |
| **Breaking change workflow** | RFC → review → deprecation window → migration | Process + tooling |
| **Security-tier contracts** | Mandatory review for crypto/auth/data contracts | CODEOWNERS + CI gate |
| **Namespace registry** | Prevent ID collisions across 500 repos | Central registry (even a YAML file in a shared repo) |
| **Compliance evidence** | Steward reports as audit artifacts | CI pipeline + artifact retention |
| **Maturity model** | Teams self-assess, get support to level up | Assessment framework |
| **Org-wide dashboard** | Executive view of contract health, test coverage, drift | Aggregated steward data |
| **Agent governance** | Which agents can modify which repos? | CODEOWNERS + permission model |

**The Breaking Change Workflow:**

This is the single most important addition at enterprise scale. When blindpipe bumps `C1-BLOBSTORE` from 2.1 to 3.0, the catalog knows that 14 repos in 6 teams depend on it. The workflow:

```
1. PROPOSE: Author creates RFC in the contract's companion file
   → architecture/CONTRACT-C1-BLOBSTORE.impl.md adds "RFC: v3.0 migration"
   → CI flags all dependents and notifies their CODEOWNERS

2. REVIEW: Affected teams review the RFC
   → Each team comments with migration estimate
   → Deadline set (e.g., 2 sprints)

3. DEPRECATE: Old version gets a sunset date
   → Contract header: <!-- DEPRECATED: 2026-06-01, SUPERSEDED BY: 3.0 -->
   → Steward warns (yellow) on deprecated deps
   → After sunset, steward fails (red) on deprecated deps

4. MIGRATE: Each team migrates on their own timeline
   → grep -rn "CONTRACT:blindpipe/C1-BLOBSTORE.2" finds all refs
   → Update CONTRACT: headers, run contract tests, PR

5. RETIRE: Old version removed after all dependents migrate
   → Catalog confirms zero references to 2.x
   → Old contract file archived
```

**This workflow is the same as rebar's existing major-version-bump process** — it just adds notification (catalog knows dependents) and a timeline (deprecation window). The tooling is a thin layer on top of what already exists.

**Security-Tier Contracts:**

Some contracts deserve more scrutiny than others. At enterprise scale, introduce tiers:

```yaml
# In contract frontmatter
security_tier: "critical"  # critical | standard | internal
```

| Tier | Review Required | Examples |
|------|----------------|---------|
| **critical** | Security team + 2 senior engineers | Auth, crypto, key management, data classification |
| **standard** | 1 team lead from affected team | API contracts, data models, protocols |
| **internal** | Normal PR review | UI components, internal utilities, theme |

This maps to CODEOWNERS rules:

```
# .github/CODEOWNERS
architecture/CONTRACT-S2-AUTH*    @security-team @tech-leads
architecture/CONTRACT-*           @architecture-team
```

**The Maturity Model:**

Not every team needs to be at Tier 4. A 3-person team building an internal tool should stay at Tier 1-2. The maturity model lets teams self-assess and get support:

| Level | What You Have | What You Get |
|-------|--------------|-------------|
| **L1: Oriented** | Cold Start Quad + CLAUDE.md | Agent orientation, basic docs |
| **L2: Contracted** | L1 + contracts + steward in CI | Interface guarantees, quality gates |
| **L3: Enforced** | L2 + CONTRACT: headers + METRICS computed | Full traceability, automated metrics |
| **L4: Connected** | L3 + catalog integration + shared ASK | Cross-team discovery, shared agent knowledge |
| **L5: Governed** | L4 + security tiers + breaking change workflow | Enterprise compliance, coordinated evolution |

**Teams choose their level.** A solo developer doing a spike stays at L1. A platform team maintaining a shared SDK needs L4-L5. There's no mandate to be at any specific level — but the catalog shows each team's level, and teams consuming your contracts might reasonably expect you to be at L3+.

**The feel:** Individual developers still feel light. Their daily workflow is: read Cold Start Quad, write code, run steward, commit. The enterprise layer is invisible unless you're changing a contract that others depend on — then the catalog notifies you of the impact, and the review workflow guides you through the change. It's like how git is lightweight for daily work but enables complex branching workflows when you need them.

## The Key Design Principle: Layers, Not Modes

The adoption curve should feel like adding layers, not switching modes. Each tier adds capabilities on top of the previous one — nothing from Tier 1 gets replaced or deprecated.

```
Tier 1 (Solo):      Files + grep + bash                          ← git only
Tier 2 (Team):      + CI + CODEOWNERS + computed metrics          ← git only
Tier 3 (Dept):      + catalog git repo + committed agent memory   ← git only
Tier 4 (Enterprise): + governance + compliance + services          ← first non-git infra
```

**Git is the only infrastructure through Tier 3.** The catalog is a git repo. Shared agent memory is a committed file. Package deps are filesystem links. Cross-repo search is grep across sibling directories. Three developers coordinating six projects need nothing beyond what they already have: git, a terminal, and their repos checked out next to each other.

A Tier 4 organization still uses `grep -rn "CONTRACT:"` to find implementations. The catalog service is an index *on top of* grep, not a replacement for it. Steward still runs locally — CI just also runs it and publishes the result. ASK still works from your terminal — shared memory just means others benefit from what the agent learned.

**This is why rebar is adoptable at any scale:** it never asks you to add infrastructure you don't need. The trigger for services isn't "you have 6 repos" — it's "you have teams who've never met each other depending on each other's contracts." Until then, git is enough.

## What Rebar Should Build (Recommendations)

### Near-term (add to rebar repo)

1. **`scripts/compute-metrics.sh`** — Generates METRICS from code (replaces manual maintenance). This is just a refactor of check-ground-truth.sh to write instead of verify. Unlocks Tier 2.

2. **Catalog aggregation script** — `scripts/catalog-collect.sh` that pulls steward-report.json from sibling repos and generates a combined index. Simple, file-based, no service needed. Unlocks Tier 3.

3. **PR template** — `.github/pull_request_template.md` with "Contracts affected:" and "Breaking changes:" sections. Unlocks Tier 2.

4. **Progressive onboarding doc** — A 2-minute quickstart that links to the 10-minute orientation that links to DESIGN.md. Unlocks Tier 2-3.

5. **Maturity model doc** — `profiles/maturity-model.md` with L1-L5 self-assessment. Unlocks Tier 4.

6. **`security_tier` field in contract template** — Add to YAML frontmatter. Unlocks Tier 4.

### Medium-term (separate tooling)

7. **Static site catalog generator** — Reads steward reports from multiple repos, generates searchable HTML. Could be a GitHub Action that runs on a schedule. Unlocks Tier 3-4.

8. **ASK shared memory via git** — Document the pattern: un-gitignore memory.md, commit it, share via push/pull. Zero new infrastructure. Unlocks Tier 3.

9. **Breaking change notifier** — When a contract version bumps, find all cross-repo dependents (from catalog) and open issues in their repos. Unlocks Tier 4.

### Long-term (platform)

10. **Contract catalog service** — Real-time API for search, dependency graph, impact analysis. Replaces the static site when you outgrow it. Unlocks Tier 4 at true enterprise scale.

11. **MCP server for contracts** — AI agents discover and query contracts via Model Context Protocol. "What contracts exist for auth?" becomes a tool call, not a grep. Unlocks AI-native enterprise workflows.

## Summary

| Question | Answer |
|----------|--------|
| **Is it too heavyweight?** | No. For what it delivers, it's remarkably light. The risk at scale isn't weight — it's entropy. |
| **Does it scale?** | The philosophy scales to any size. Git is sufficient infrastructure through ~50 devs/40 repos. Services only needed at true enterprise scale. |
| **Is it adoptable at large org?** | Yes, via phased adoption (L1-L5). Teams choose their level. No big-bang migration. |
| **What's the transition cost?** | Tier 1→2: 1 day. Tier 2→3: 1-2 weeks. Tier 3→4: 1-2 months. Each tier delivers independent value. |
| **What's the killer feature at each tier?** | T1: Cold Start Quad. T2: CI enforcement. T3: Contract catalog. T4: Breaking change workflow. |
| **Does a solo dev feel burdened?** | No. They use exactly what helps them and nothing more. Enterprise layers are invisible until needed. |
